# Регламент разработки и качества

## Цель
Обеспечить предсказуемые изменения без регрессий и сохранить стабильность ключевых пользовательских сценариев.

## Обязательные принципы
1. Сначала безопасность и корректность.
2. Затем обратная совместимость.
3. Затем читаемость и поддерживаемость.
4. Затем минимальность диффа.

## Рабочий процесс для любой задачи
1. Зафиксировать ожидаемый эффект для пользователя.
2. Найти минимальную точку правки.
3. Проверить связанные импорты/экспорты и соседние зависимости.
4. Внести атомарное изменение.
5. Добавить обработку ошибок в новые асинхронные ветки.
6. Выполнить минимальную валидацию.
7. Описать результат, риски и ручные шаги проверки.

## Требования к коду
- Не добавлять тяжелые зависимости без явного запроса.
- Не переименовывать публичные сущности без необходимости.
- Не выносить большой рефакторинг в задачу, где нужен локальный фикс.
- Не оставлять временные отладочные конструкции.

## Требования к обработке ошибок
- Любая новая async-ветка обязана иметь predictable error path.
- Любые внешние данные обязаны проходить валидацию на пустые и частичные состояния.
- UI должен иметь согласованные состояния loading/empty/error/success.
- Ошибка не должна приводить к необратимой потере пользовательского состояния.

## Особые правила по зонам

### Расписание
- Проверять переключение день/неделя.
- Проверять выбор курса/группы и подгрузку данных.
- Проверять оффлайн-сценарий и fallback на кэш.

### Новости
- Проверять фильтрацию и дедупликацию.
- Проверять корректность сортировки.
- Проверять поведение при повторных обновлениях.

### Карта
- Проверять фильтры и отрисовку маркеров.
- Проверять fallback/ошибки карты.

### Настройки
- Проверять сохранение и восстановление значений.
- Проверять, что изменение настройки влияет на соответствующий экран/сервис.

## Минимальная валидация перед сдачей
- Запуск: `npm run start`.
- Проверка измененного пользовательского сценария вручную.
- Проверка отсутствия синтаксических и импортных ошибок в затронутых файлах.

## Расширенная валидация (если релевантно)
- Уведомления: отсутствие дублей, корректный тайминг.
- Фоновые задачи: отсутствие лишних частых запусков.
- Хранилище: корректное чтение старых значений.
- Ассеты: корректные пути и отображение на основных экранах.

## Формат отчетности по задаче
- Что сделано.
- Какие файлы изменены.
- Почему выбран этот подход.
- Какие риски остаются.
- Что проверить вручную.
